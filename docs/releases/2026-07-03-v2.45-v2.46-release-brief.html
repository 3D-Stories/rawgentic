removed BMAD integration & disabledSkills v2.45.0 · #124

The BMAD coexistence layer is gone: the _bmad/ detection probe in switch/setup, the per-project skill-preference UI, the disabledSkills check in every workflow skill, and the CLAUDE.md routing template.

If your workspace still has bmadDetected or disabledSkills fields: they are now ignored — nothing errors, nothing to migrate. Skills previously blocked by disabledSkills (e.g. update-docs) are active again everywhere.

new modelRouting — route subagent fleets to cheaper models v2.46.0 · #128

Rawgentic workflows dispatch subagent fleets at seven sites (3 design judges, 2 per-task reviewers, 3 pre-PR reviewers per WF2 run, codebase gather, background memorization). Until now every one inherited your session model — premium tokens on work a cheaper strong model handles. Now each role is routable, per project:

// .rawgentic_workspace.json — your project's entry
"modelRouting": {
  "review": "opus",          // all judge/reviewer fleets
  "analysis": "sonnet",      // codebase gather + memorization
  "implementation": "opus"   // opt-in Step 8 task delegation
}
  • Default-off: absent block or role = inherit — subagents use the session model, byte-identical to v2.45.
  • Fail-open: malformed/unknown values warn and resolve to inherit. Routing is a cost knob, never a gate — it cannot block a run.
  • Soft opus floor: routing review below opus (explicit sonnet/haiku) warns but still applies — your call per project.
  • Configure via /rawgentic:setup (new Step 2f) or hand-edit the workspace JSON.

Step 8 implementation delegation (the big token lever)

With implementation set, WF2 Step 8 hands each validated plan task to a subagent on that model instead of executing inline — serial, one task at a time, each ending committed. A clean-state boundary protects every task: pre-task state is recorded, and any failed/vacuous task-agent is rolled back before the one inline retry, so a retry never runs on a half-mutated tree. Implementation is the largest token block of a WF2 run; this moves it off your premium session model — the design and plan (already critiqued by that point) steer the executor.

new peerConsult — Codex as a peer designer, not a reviewer WF13 · v2.46.0 · #128

WF5 already lets Codex critique your artifacts. WF13 lets it propose: an independent, different-model design for the same problem.

Standalone — works today

/rawgentic:peer-consult docs/design/feature.md

Sends the problem artifact to Codex as a peer senior engineer; writes its proposal (approach, key decisions, risks, sketch) to docs/reviews/peer-<slug>-<date>.md. Report-only — never edits your artifact. Requires the Codex CLI installed + authenticated (same as WF5); artifact text leaves the machine (egress warning shown).

WF2 integration — opt-in, blind both ways

"peerConsult": {
  "enabled": true,
  "workflows": ["implement-feature"]
}

At the WF2 design step, Codex drafts its proposal in the background while Claude drafts its own — neither sees the other's until Claude's draft is on disk — then Claude synthesizes best-of-both with provenance recorded. Codex failure never blocks the step. Configure via setup Step 2g.

WF5 vs WF13 in one line: WF5 tears a finished artifact down (evaluative gate); WF13 proposes an independent alternative up front (generative design). Both can be on at once — they run at different steps.
