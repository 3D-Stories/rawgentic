# Configuration Reference

## Overview

Every rawgentic project has a `.rawgentic.json` file at its root. This file is the
single source of truth for project metadata consumed by all workflow skills.

- Generated by `/rawgentic:setup`, which auto-detects the project's tech stack,
  testing, CI, infrastructure, and more.
- All workflow skills load this file via the config-loading protocol (see below).
- Full annotated schema with example values: `templates/rawgentic-json-schema.json`.

## Design Principles

**Config-driven.** Workflow skills never probe the filesystem for information that
should be in the config. If it is detectable at setup time, it belongs in
`.rawgentic.json`.

**Single source of truth.** No duplication between `.rawgentic.json` and CLAUDE.md.
Config holds structured data; CLAUDE.md holds prose conventions and instructions.

**Capabilities object.** Every workflow skill derives a boolean `capabilities` object
from the config at startup. The exact flags are:

| Flag | Derivation |
|------|-----------|
| `has_tests` | `config.testing` exists AND `config.testing.frameworks.length > 0` |
| `test_commands` | `config.testing.frameworks[].command` |
| `has_ci` | `config.ci` exists AND `config.ci.provider` exists |
| `has_deploy` | `config.deploy` exists AND `config.deploy.method` exists and != `"manual"` |
| `has_database` | `config.database` exists AND `config.database.type` exists |
| `has_docker` | `config.infrastructure` exists AND `config.infrastructure.docker.composeFiles.length > 0` |
| `project_type` | `config.project.type` |
| `repo` | `config.repo.fullName` |
| `default_branch` | `config.repo.defaultBranch` |

Skills adapt their behavior based on these flags -- for example, a project without
tests uses implement-verify mode instead of TDD, and a project without CI skips the
CI verification step.

**Learning config.** Skills can update `.rawgentic.json` when they discover new
project information during execution:

- Append to arrays (e.g., a newly discovered test framework).
- Set fields that are currently null or missing.
- Never overwrite existing non-null values without asking the user.
- Always read the full file, modify in memory, and write back.

## Core Sections

| Section | Purpose | Primary consumers |
|---------|---------|-------------------|
| `project` | Name, type (`application`, `library`, `infrastructure`, etc.), description. | All skills (for project-type-specific behavior). |
| `repo` | Provider, full name, default branch. | All skills (for `gh` CLI commands and branching). |
| `techStack` | Array of technologies (languages, frameworks, databases). | setup, create-issue, implement-feature, optimize-perf, update-deps. |
| `testing` | Test frameworks with commands, config files, and test directories. | implement-feature, fix-bug, refactor, optimize-perf, update-deps, update-docs, security-audit, incident. |
| `database` | DB type, CLI tool, name, user, container, migrations directory and pattern. | implement-feature, fix-bug, optimize-perf, incident, security-audit. |
| `services` | Array of service objects (name, type, framework, port, entry point, health check). | create-issue, implement-feature, fix-bug, incident, security-audit, optimize-perf, update-docs. |
| `infrastructure` | Hosts (SSH targets) and Docker compose files. | implement-feature, fix-bug, refactor, incident, optimize-perf, update-deps, security-audit. |
| `deploy` | Method (`compose`, `ssh`, `script`, `manual`), command, target host. | All SDLC skills (for merge-and-deploy steps). |
| `security` | Auth mechanisms, validation library, data channels. | security-audit (primary), create-issue, implement-feature. |
| `ci` | Provider and workflow directory. | All SDLC skills (for CI verification gate). |
| `formatting` | Tool, config file, and format command. | implement-feature, fix-bug, refactor (for code style compliance). |
| `documentation` | Primary doc files and format. | update-docs (primary), setup. |
| `custom` | Free-form object for project-specific metadata (e.g., security debt). | Any skill; security-audit checks for known debt items. |

## Versioning

The top-level `version` field tracks the config schema version. Current version: **1**.

Skills check this field at startup:
- Missing or `> 1`: warn the user about a version mismatch.
- Equal to `1`: proceed normally.

## Config-Loading Protocol

Every workflow skill executes the following protocol before any workflow steps.

### 1. Determine the active project (3-level fallback)

| Level | Source | Behavior |
|-------|--------|----------|
| 1 | **Conversation context** | If `/rawgentic:switch` was used in this session, use that project. |
| 2 | **Session registry** | Read `claude_docs/session_registry.jsonl`, grep for the current session ID, use the project from the most recent matching line. |
| 3 | **Workspace default** | Read `.rawgentic_workspace.json` from the Claude root directory. Use the project where `active == true`. If multiple are active, stop and ask the user to run `/rawgentic:switch`. |

### 2. Load the config file

Read `<activeProject.path>/.rawgentic.json`. Path resolution: if `activeProject.path`
is relative (e.g., `./projects/my-app`), resolve it against the Claude root directory.

### 3. Build the capabilities object

Derive the boolean flags listed above from the parsed config.

### Error behavior

| Failure | Response |
|---------|----------|
| `.rawgentic_workspace.json` missing | Stop. Tell user to run `/rawgentic:new-project`. |
| `.rawgentic_workspace.json` malformed | Stop. Tell user the workspace file is corrupted. |
| No active project at any level | Stop. Tell user to run `/rawgentic:new-project` or `/rawgentic:switch`. |
| `.rawgentic.json` missing | Stop. Tell user to run `/rawgentic:setup`. |
| `.rawgentic.json` malformed JSON | Stop. Tell user config is corrupted; suggest `/rawgentic:setup`. |
| `config.version` missing or `> 1` | Warn user about version mismatch but continue. |
