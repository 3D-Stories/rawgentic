## Summary

The artifact designs a sibling GLM backend and dual-backend dispatch with fail-loud result handling. The largest risks are an unverified hard-timeout guarantee, an incomplete partial-result protocol, and platform compatibility that may remain unverified at merge.

## Findings

### 1. [High] correctness · high confidence — File changes item 1 — `run_glm_consult` both-mode contract

> Consumers (WF13 present step; WF2
>      Step 3 embedded synthesis) derive the EXPECTED proposal set from the exit code:
>      exit 0 → both files must exist and parse (the empty-proposal-marker guarantee +
>      per-backend write-failure rule make a missing expected file impossible without
>      that backend counting failed); exit 5 → the successful backend's file.

Exit 5 identifies that exactly one backend succeeded but does not identify which backend. File existence cannot safely disambiguate it because failed consult runs write empty-proposal markers. An embedded consumer can therefore select the failed backend's marker or a stale file instead of the successful proposal.

**Recommendation:** Add a machine-readable both-mode result manifest containing each backend's status and output path, and require WF13 and WF2 Step 3 to select proposals exclusively from entries whose status is `success`. Do not derive the expected file from exit 5 or existence checks.
**Ambiguity:** The artifact does not define how consumers learn which backend succeeded on exit 5.

### 2. [High] feasibility · high confidence — File changes item 1 — `_glm_chat` timeout design

> **Test-layer honesty (pass-2 A12):** an injected fake
>      client has no httpx, so layer 1 is NOT fake-testable — CI exercises layer 2
>      only, via a TRICKLING iterator (sleeps between chunks past the deadline; a
>      literally-blocked fake `next()` would hang the suite). Layer 1 is verified by
>      the pre-merge live smoke.

Layer 1 is the only mechanism capable of interrupting a blocked `next(stream)`, but neither CI nor an ordinary successful live smoke exercises that condition. If the SDK constructor timeout does not cover streamed socket reads, a stalled response hangs the host workflow indefinitely instead of producing exit 3.

**Recommendation:** Change the Platform / external dependencies verification gate to require a targeted stall test using the exact supported zhipuai version and a server that sends headers then blocks the response body. If that cannot prove interruption, execute each attempt in a killable worker process with an outer hard deadline.

### 3. [Medium] ambiguity · high confidence — File changes item 1 — both-mode sidecar consumer contract

> on exit 0 it reads
>      BOTH the exact path and the `-glm` sibling and merges (dropping the sibling
>      would silently lose every glm finding at the highest-value gate).

“Merges” does not define ordering, duplicate handling, severity conflicts, or how provenance is retained after combining the legacy GPT schema with GLM findings. Different implementations can produce different gate inputs or double-count equivalent findings.

**Recommendation:** Replace “merges” in the Consumer contract with a deterministic algorithm: attach provenance from file identity, normalize both lists, deduplicate by `(evidence, location, category)`, retain the highest severity on collisions, and emit a stable severity/backend/order sort.
**Ambiguity:** The merge operation has no specified semantics beyond reading both files.

### 4. [Medium] feasibility · high confidence — File changes item 1 — GLM prereq helpers

> `glm_sdk_available()` (deferred
>      `importlib.util.find_spec("zhipuai")` — never imports at module load),

The prereq gate checks only that a module is discoverable, not that its installed version satisfies the stated `zhipuai>=2.1.5` call-shape floor. An older installation will be reported ready and fail only after invocation reaches client construction or `create()`.

**Recommendation:** Change `glm_sdk_available` into a compatibility check that reads `importlib.metadata.version("zhipuai")`, rejects versions below 2.1.5 with upgrade guidance, and reports the detected version in diagnostics.

### 5. [Medium] feasibility · high confidence — Platform / external dependencies — install lifecycle and fallback

> Fallback if the key is genuinely unreachable mid-run (pass-2 A7 wording):
> the PR ships **explicitly NOT-live-verified** — deferred-to-target with the #138
> recording discipline (plan task line, PR `## Deferred verification` section,
> run-record), naming the likeliest-wrong claim (exact kwarg acceptance of
> `thinking=`/`extra_body=` by the installed zhipuai version).

The fallback permits merging while a load-bearing external call shape is explicitly unverified under this project's installed SDK and endpoint. Documentation of the deferral surfaces the uncertainty but does not establish that the selectable GLM backend can execute in the target configuration.

**Recommendation:** Change the release gate so GLM remains explicitly disabled or experimental unless an attached smoke record proves the exact constructor, `thinking=`, `extra_body=`, streaming, and response-format call against the supported SDK version. Enable the documented backend only after that record exists.

### 6. [Medium] internal-consistency · high confidence — File changes item 1 — report renderers; Backward compatibility

> - `render_report_md` / `render_consult_md`: meta gains `backend`; Reviewer line
>      renders `Codex (model …, reasoning effort …)` or `GLM (model glm-5.2, reasoning
>      effort …)` (AC7). Absent meta key → existing Codex wording (back-compat).

Adding `backend` to rendered metadata changes GPT report bytes, contradicting the later claim that default GPT users are byte-for-byte unaffected. Existing report snapshots or strict metadata consumers can break without opting into GLM.

**Recommendation:** Change the renderers to omit the `backend` metadata field for single-backend GPT runs and add whole-report golden tests proving legacy GPT output is byte-identical. Otherwise remove the byte-compatibility claim and document the format change.

### 7. [Medium] internal-consistency · high confidence — File changes item 1 — `AdversarialReviewConfig`; File changes item 2 — backend resolution

> and every entry point
>      (`prereq`, `review`, `consult`, the `backend` subcommand) refuses with exit 2
>      and a message naming the rejected value BEFORE any provider call

This unconditional refusal conflicts with the later rule that an explicit valid `--backend` skips config resolution entirely, allowing the same entry points to proceed despite an invalid configured backend. Implementers could enforce either precedence and produce inconsistent egress behavior.

**Recommendation:** Qualify this configuration rule to say “every entry point that resolves backend from configuration,” then define and test a single precedence table covering explicit CLI backend, valid config, absent config, disabled config, and invalid config.
**Ambiguity:** Two sections assign different effects to an invalid config when an explicit backend is present.

### 8. [Medium] security · medium confidence — File changes item 1 — GLM prereq helpers and endpoint handling

> `glm_base_url()` (`ZHIPUAI_BASE_URL` → `GLM_JUDGE_BASE_URL` → default
>      `https://api.z.ai/api/coding/paas/v4`, the Coding Plan subscription endpoint, AC9).

The design accepts an environment-supplied base URL but specifies only sanitized logging, not URL validation. A mistaken or malicious `http://` override can transmit the API key and artifact without TLS, while an unintended HTTPS host receives both under an apparently valid GLM invocation.

**Recommendation:** Add validation in `glm_base_url`: require HTTPS, reject userinfo/query/fragment components, and allow non-z.ai hosts only behind an explicit custom-endpoint opt-in that is named in the egress consent message.
