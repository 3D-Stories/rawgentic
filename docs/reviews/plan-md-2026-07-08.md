## Summary

The artifact is a sequenced implementation plan for delegated read digests, with validation, workflow wiring, an A/B experiment, and docs/release updates. The main risks are unproven dependence on external model/subagent capabilities and acceptance criteria that are not fully executable from the plan text alone.

## Findings

### 1. [High] feasibility · high confidence — Task 3: A/B experiment run

> A-arm: subagent per artifact, full raw diff + questionnaire. B-arm: sonnet reader → index → validate_index (Task 1 code) → consumer answers questionnaire via index + targeted reads. 3 live adversarial probes (fabricated quote / dropped file / vacuous) must REJECT. Score per design rubric (F1 mechanical, blinded opus judge for q2/q4/q5/q6, unlabeled random order).

Task 3 depends on platform/model capabilities (`subagent`, `sonnet reader`, and `blinded opus judge`) but does not include any proof that these APIs/tools work under this project's real configuration. If those model routes, permissions, quotas, or subagent contracts are unavailable or different, the A/B experiment cannot produce valid owner-acceptance evidence.

**Recommendation:** Change Task 3 to add a required preflight before the A/B run: execute one minimal configured subagent call, one sonnet-reader call, and one opus-judge call under the project’s actual config; record model identifiers, tool/config source, permissions, token accounting availability, and failure handling in `docs/planning/2026-07-08-314-ab-results.md`; block the experiment if any preflight fails.

### 2. [Medium] ambiguity · high confidence — Task 3: Pass criteria

> Pass: B<A tokens AND |pooled delta| ≤5% AND no artifact >10% below.

The acceptance bar does not define how `pooled delta` or `artifact >10% below` are calculated across F1 and judge-scored questions. A reasonable implementer would have to stop and ask whether the delta is absolute score, relative score, weighted by question/artifact, macro-averaged, micro-averaged, or judge-normalized, which can change pass/fail outcome.

**Recommendation:** Change Task 3 `Pass` to specify the exact formulas: define the score scale, per-question weighting, per-artifact aggregation, pooled aggregation, and whether the 10% threshold is absolute percentage points or relative percent below A-arm.
**Ambiguity:** The metric names are not self-defining and the plan does not include formulas.

### 3. [Medium] ambiguity · high confidence — Task 1: VERIFY

> VERIFY: pytest tests/hooks/test_plan_lib_index.py -q green; full suite delta.

`full suite delta` is not an executable verification command or pass/fail criterion. This makes Task 1 verification incomplete because an implementer cannot know which full suite command to run, what baseline to compare against, or what amount of delta is acceptable.

**Recommendation:** Change Task 1 `VERIFY` to name the exact full-suite command and acceptable result, for example `pytest -q` must pass with no new failures, plus any expected test-count changes explicitly listed.
**Ambiguity:** The phrase implies comparison but omits the baseline and acceptance threshold.

### 4. [Medium] ambiguity · high confidence — Task 4: VERIFY

> VERIFY: pytest tests/test_workflow_diagram.py + full suite + both pylint lanes.

`full suite` and `both pylint lanes` are not specified as concrete commands. The release task could be implemented and verified inconsistently because the plan does not identify the actual lint entrypoints or required pass criteria.

**Recommendation:** Change Task 4 `VERIFY` to list the exact commands for the full test suite and both pylint lanes, and state that all must pass before the task is complete.
**Ambiguity:** The plan names categories of checks but not the actual commands or acceptance outputs.
