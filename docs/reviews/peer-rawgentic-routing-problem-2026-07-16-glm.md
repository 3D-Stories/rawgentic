# Peer Consult — .rawgentic-routing-problem.md

- Date: 2026-07-16
- Reviewer: GLM (model glm-5.2, peer designer)

## Approach

Adopt Candidate C — retune the existing 3-role values now (no schema migration), and reserve per-phase keys only if a future bench proves >=2 distinctions the role model cannot express. Concretely: driver = opus-4-8 (owner's inline design/plan/ship review sessions), review = fable-5 with opus-4-8 quota fallback, analysis = sonnet-5, implementation ceiling = opus-4-8 with select_impl_model continuing to down-route standard/trivial build tasks to sonnet. This captures the only three statistically real edges (fable review +4, sonnet ship +3, fable/opus intake parity) without inventing a 6-key phase map the n=6 data cannot defend. Fable is farmed to subagents only on review dispatch points, never on the high-frequency build path; when fable quota is exhausted, review fails open to opus (never below, per the standing review soft floor).

## Key decisions

- Driver = opus-4-8. The owner runs intake/design/plan/ship inline; sonnet's collapse floors on the harder brief (design 46, plan 28) and 5/6 gate pass-rate are disqualifying for the interactive seat even though its medians are competitive. Sonnet's ship edge is real but irrelevant to the driver because ship-phase work is dispatchable to subagents and the edge is consumed there, not in the owner's session.
- Review = claude-fable-5. The +4 review edge (sd~2) is the one Claude-family quality gap that clears noise, and review gates everything downstream (Steps 4, 8a, 11). Accept the 1.6x API-equivalent cost because on Max subscription the marginal cost is usage-window burn, and review volume is bounded (1 + high-risk-task-count + 1) rather than per-token. Quota-exhaustion fallback = opus-4-8, enforced by an explicit fable->opus chain so fail-open never drops below the review soft floor.
- Analysis = claude-sonnet-5. No statistically distinguishable Claude gap on analysis-adjacent phases, sonnet is cheaper, and the intro pricing makes it ~1/3 cheaper still through 2026-08-31. Tail risk is acceptable here because analysis is not the driver's inline gate.
- Implementation ceiling = opus-4-8; keep select_impl_model unchanged. Build is exactly the close-score/big-cost-gap case the owner wants forked: opus 77 vs sonnet 76 (gap 1, pooled sd 7.6) but opus 1.74x cost. The existing riskLevel+complexity fork already routes complex/high-risk build tasks to the opus ceiling and standard/trivial tasks to sonnet — reuse it rather than adding a build-phase key.
- Complexity fork keys off both issue.complexity and per-task riskLevel, exactly as select_impl_model already does. Do not introduce a single-axis fork; the dual signal is fail-closed and already wired (plan_lib riskLevel, WF2 Step 2 complexity).
- Provenance: stamp modelRouting values with bench_id (#14) and a one-line rationale per role in .rawgentic_workspace.. Refresh is human-in-the-loop, not an auto-tuner: a new bench report triggers a runbook that only moves a value when (median gap > pooled sd) AND (worst-cell floor stays above the role's acceptable floor). Otherwise values hold.
- Defer per-phase schema (Approach A). The data supports 3 distinctions, not 6; intake/design/plan opus-vs-sonnet are all inside noise, and build is already handled by the implementation fork. Add per-phase keys only when a future bench shows >=2 new real distinctions the 3-role model cannot express.

## Risks

- Fable quota is historically tight and review dispatch is the highest-frequency Claude path (Step 8a scales with high-risk task count). Under a high-risk-heavy plan, fable could exhaust mid-run; the opus fallback must be verified to fire atomically and not stall the in-flight agent (session-limit deaths).
- Single benchmark, n=6 cells/phase/model. Sonnet's 46/28 collapse cells could be noise on a different brief set; equally, fable's review edge could shrink. Design must not overfit — hence the gap>sd + floor rule and deferred per-phase schema.
- Sonnet analysis on harder briefs (brief a) carries the same tail-risk family as sonnet plan; analysis is lower-stakes but not zero-stakes. If a future bench shows analysis tail, promote analysis to opus.
- Max ~3 concurrent Claude subagents: Step 8a per-task review fan-out must be concurrency-clamped; fable review agents competing with build subagents can trigger session-limit deaths. Need an explicit semaphore, not just the quota guard.
- Cost cross-engine comparison (gpt rows est@12/M) is explicitly soft; do not let the terra/sol gpt numbers drive routing — they only inform that Claude-family gaps are small, not that gpt should displace Claude roles.
- Sonnet intro pricing expires 2026-08-31; the cost case for analysis=sonnet weakens but the role assignment is still defensible on the no-real-gap argument.
