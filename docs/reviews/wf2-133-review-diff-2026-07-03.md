## Summary

The diff adds an opt-in whole-issue delegation mode plus a receipt validator intended to fail closed before trusting subagent-built commits. The main risk is that the validator binds only declared plan tasks, while its branch-diff equality accepts arbitrary extra receipt entries as if they were legitimate task ownership.

## Findings

### 1. [High] security · high confidence — hooks/plan_lib.py validate_build_receipt Rule 4

> +        for v in files_per_task.values():
> +            if isinstance(v, list):
> +                union |= {x for x in v if isinstance(x, str)}

The staging-discipline check builds its accepted file union from every `files_per_task` entry, including keys that are not real plan task ids. Because the earlier sha/file binding loop only iterates `plan_tasks`, a builder can add an extra fake task entry claiming an unplanned changed file; that file is then no longer `undeclared`, but no corresponding sha is required, no commit existence/lineage/distinctness is checked, and no per-commit file binding is validated. The concrete failure outcome is that a receipt can validate with branch changes outside the approved plan task set.

**Recommendation:** In `validate_build_receipt`, reject any `task_shas` or `files_per_task` keys outside `{task.id for task in plan_tasks}` before computing the union, and compute Rule 4's union only from plan task ids after their entries have passed type checks.

### 2. [Medium] completeness · high confidence — hooks/plan_lib.py validate_build_receipt initialization

> +    normalized["promoted_task_ids"] = []
> +    if not isinstance(receipt, dict):
> +        return (False, ["receipt is not a dict"], normalized)

The contract says promoted task ids are surfaced even on rejection where derivable, but this early non-dict rejection returns only an empty normalized object. For malformed but still derivable receipt data this is acceptable, but for a non-dict receipt there is no way to derive promotions, so the implementation is fine; however, the documentation does not distinguish non-derivable malformed receipts from derivable partial receipts. A reasonable implementer/operator could expect Step 8a promotion scheduling to still happen on every rejected receipt.

**Recommendation:** In the docstring and `whole-issue-delegation.md`, change the promotion guarantee to explicitly say promotions are surfaced on rejection only when `receipt` is a dict and `promotions` is a list of objects with string `task_id`.
**Ambiguity:** The code behavior is defensible, but the written contract is broader than the implementation for malformed receipt shapes.
