Verdict

The architecture holds. Pattern-not-skill was the right cut — the driver structurally cannot weaken WF2 because it never owns a WF2 step; the committed-contract vs disk-persisted-state split is honest about the gitignored claude_docs/; and the narrow-helper scope discipline (ship only what the ACs make testable, defer the state-machine validator) is exactly the restraint #134 asked for. The defects are at the edges: one genuine design contradiction inherited from #134's own ambiguity (F1), a dependency parser whose in-slot review fixes traded over-capture for a common-phrasing under-capture (F2/F3), and three spec gaps a first real campaign would trip on (F4–F6). Nothing here invalidates the merge; F1 needs an owner call before the first pr_open-knob campaign runs.

Findings all executed against d7ea584

F1 · Serial-active invariant contradicts the deps_satisfied_by="pr_open" knob High · design consistency Owner fork

validate_driver_state rejects any state with more than one issue in {in_progress, pr_open} — but the #163-AC3 knob exists precisely to let a dependent start while its dependency sits pr_open (headless stacked-PR flow: "advance the queue on PR creation"). The moment a knob-enabled campaign starts issue B with issue A at pr_open, the state is invalid. Both sides are test-locked: test_validate_driver_state_serial_active_invariant asserts the combo fails while test_next_ready_issue_pr_open_knob asserts the advance that creates it.

# executed probe — the state the knob is designed to produce:
state = {issues: [#1 pr_open, #2 in_progress (deps [1]), #3 queued]}
validate_driver_state(state)
→ (False, ['at most one issue may be in_progress/pr_open (serial driver); found 2: [1, 2]'])

Root cause: design #134 itself is torn — L58 calls pr_open "the terminal state for a headless driver," L97 says the headless driver "advances the queue on PR creation." The in-slot Codex F2 fix codified L58; the #163 knob codifies L97.

Recommendation: relax the invariant to in_progress ≤ 1 (one build at a time; pr_open may accumulate awaiting human merge) — that matches the knob and the "advance on PR creation" prose. Alternative: keep the strict invariant and document the knob as resumption-reconciliation-only. Either is a two-line change + doc sentence; the fork is the owner's.

F2 · Parser silently drops the most common natural phrasing Medium · correctness

_DEP_LIST_RE demands #N immediately after the phrase (modulo colon/whitespace), so any interposed noun kills the whole capture — a missing ordering edge, i.e. the premature-start mis-ordering the DAG exists to prevent, and silent (empty list, no error).

parse_depends_on("Depends on issue #10.")            → []   # expected [10]
parse_depends_on("Blocked by issues #10 and #20.")   → []   # expected [10, 20]
parse_depends_on("depends on the epic #30")          → []   # expected [30]

Root cause: the in-slot Codex High fix (sentence-boundary grammar) correctly killed prose over-capture ("Depends on #10. See #20" → [10]) but traded it for this under-capture, unmitigated. Fix: allow an optional (?:the\s+)?(?:issues?|prs?|epics?)\s+ before the #N list; add the three probes above as regression tests.

F3 · Grammatical negations are dead letters — modal negations over-capture Med-Low · correctness

_NEG_BEFORE_RE requires the negation word immediately before the phrase, so its modal entries (cannot, can't, won't, never) only fire on ungrammatical text — real English puts "be" between them and the phrase.

parse_depends_on("cannot be blocked by #5")          → [5]  # a statement of NON-dependency
parse_depends_on("This will never be blocked by #5") → [5]
parse_depends_on("This is not really blocked by #5") → [5]  # "immediate-only" is documented, but…

Fix: permit an optional (?:be|get)\s+ between negation and phrase. The "not really" case can stay (documented immediate-negation scope) — the modals are the actual bug, since they're listed in the regex yet can practically never match.

F4 · String-typed depends_on silently drops the edge Med-Low · robustness

A hand-edited live state file (the resumption substrate is hand-maintained JSON by design) with "depends_on": ["148"] imposes no edge in either topo_sort_issues or next_ready_issue — the string is filtered as "external" and assumed satisfied. Executed: next_ready_issue returned the dependent before its dependency. validate_driver_state catches the type — but nothing in code or doc mandates validating on state-file load; validate_campaign_start is start-only.

Fix: one doc sentence in §Resumption ("validate on every load, not just start") — or harden _in_queue_deps to raise on non-int, matching the module's fail-closed posture.

F5 · The "ledger" is referenced twice, defined nowhere Med-Low · spec gap

Doc L124 ("park, surface in ledger") and L224 ("note the resume-after-window-reset time in the ledger") both route information to a ledger with no location, format, or owner. Design #134 listed the "alignment ledger" as one of the hand-rolled artifacts the pattern was meant to formalize — it lost its definition in the build. First campaign to DEFER something has nowhere specified to surface it.

Fix: define it — simplest honest answer: the ledger is the state file's per-issue fields + a dated notes[] array (schema already tolerates extra keys via additionalProperties: true), or the campaign's session-notes section; one paragraph either way.

F6 · "Parked" is two different states Low · spec ambiguity

The DEFER table says a dependent of a deferred issue is parked with deferred_reason: cross-issue-dependency (status → deferred); the advance rule + code treat parking as skip-while-queued (status untouched, no reason ever recorded — next_ready_issue just passes over it). Same event, two contradictory state outcomes. If the orchestrator follows the table, the dependent leaves queued and the skip logic is dead code; if it follows the code, the taxonomy row's reason is never set.

Fix: pick one — recommend transient skip (code behavior) with the taxonomy row reworded to "implicitly parked while queued; mark deferred only if the dependency is abandoned."

F7 · Epic body through parse_depends_on = all children become "dependencies" Low · footgun

The same - [ ] #N syntax serves two roles: epic queue derivation and dependency declaration. Executed: an epic body with two task-list children returns [201, 202] as dependencies. The doc never warns that the epic body must not be dependency-parsed — a driver that naively parses every issue body inverts the epic into a self-dependency knot.

Fix: one doc sentence in §Epic anchor: "never run parse_depends_on on the epic body itself."
