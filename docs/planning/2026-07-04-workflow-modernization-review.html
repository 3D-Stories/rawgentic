AC2 /goal integration Adopt — Step 1b goal guard

Confirmed: /goal is GA and headless-capable; 4,000-char condition; skills cannot set it — they construct the text and hand it over.

The flow

Extract numbered ACs from the validated issue → one user verification → emit compact goal text: "Issue #N done: verified ACs met, PR open with green CI, run-record persisted — or a blocker is posted via the ERROR protocol."

The escape clause is the design

The "or a blocker is posted" disjunct lets a legitimately-blocked run satisfy the goal honestly — no lying, no manual clear in the common failure path. Goal is a condition, never instructions (re-injection burn).

Headless

wf1-created-labeled issues use their pre-approved ACs verbatim; unlabeled issues skip the guard. Driver or Action sets the goal at session start. PR-terminal wording always.

AC3 Multi-issue runs Extend #148 — add dependency DAG

Fresh WF2 invocation per issue (the shipped #134 design) is the fix for "first faithful, rest loose." This review endorses it unchanged and adds one layer.

Dependency-ordered execution

Parse depends on #N / task-lists / GH relationships → issues[].depends_on in the queue file (schema v2) → topo-sort at campaign start → halt fail-closed on cycles. Advance only when deps are merged (policy knob for pr_open).

Deferred deps park dependents

cross-issue-dependency DEFER reason; the loop continues with independent issues — never stalls the backlog.

Campaign-level /goal + compaction

One goal wraps the driver session (one goal per session); compaction between issues is the proven pattern from 131–140.

Epic anchor — offered, not forced

Epic issue = human-facing intent + progress mirror (task-list checkboxes, STATUS/QUESTION channel, DAG input); the state file stays the machine source of truth, mirrored one-way. Optional interactively, mandatory headless — an unattended run needs somewhere to talk to its humans.
