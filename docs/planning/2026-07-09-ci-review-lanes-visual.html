Claude CI Review Lanes

Two GitHub Action lanes run Claude's built-in reviews on every pull request to main: claude-code-review.yml and claude-security-review.yml. Both are advisory — a red check never blocks a merge — and both carry the honest-signal contract: green means the review actually ran and succeeded; red means it did not run.

ADVISORY — not required checks green = reviewed red = NOT reviewed

01

At a glance

code-review  claude-code-review.yml

Runs the built-in /code-review over the PR diff — a second, cross-checking reviewer alongside WF2's hand-rolled Step 11 panel. Issues #196 (reopens #162), #233.

Triggerpull_request → main
types: [opened, synchronize, reopened, ready_for_review]
Draft gateYES  if: github.event.pull_request.draft == false
Runsprompt: "/code-review" via anthropics/claude-code-action@v1 (SHA-pinned)
Findings landInline PR comments + GITHUB_STEP_SUMMARY line
Telemetryreviewer_kind: builtin_code_review — the #162 A/B arm

Why ready_for_review: default trigger types don't include it — a PR marked "Ready for review" with no later push would never get a builtin review. WF2 drafts + un-drafts PRs on its suspend/resume path.

security-review  claude-security-review.yml

Runs the built-in /security-review over the PR; confirmed findings posted as inline PR review comments (the action's classify_inline_comments mechanism). Issues #166, migrated #195, #233.

Triggerpull_request → main
default types (opened, synchronize, reopened)
Draft gateNO  runs on draft PRs too
Runsprompt: "/security-review" via anthropics/claude-code-action@v1 (SHA-pinned)
Findings landInline PR review comments + GITHUB_STEP_SUMMARY line
TelemetryClean runs count toward the 10-PR promotion tally

Complements, never replaces, the local Step 11.5 tool scanners (hooks/security_scan.py: secrets, CVEs, SAST, IaC). This lane is the semantic pass.

BONUS SECOND OPINIONS. These lanes sit on top of WF2's in-workflow gates — Step 11 (hand-rolled multi-agent code review) and Step 11.5 (security_scan.py fail-closed scan) remain the primary gates. The lanes replace nothing; they are most valuable on human-opened or non-Claude PRs that never went through WF2 at all.

Shared permissions on both jobs: pull-requests: write (post review comments) · contents: read (checkout) · id-token: write (the action's default auth mints a short-lived Claude GitHub-App token via OIDC token-exchange — no static PAT). Runner: ubuntu-latest. Neither job carries continue-on-error — by design (see below).

03

The honest-signal contract (#233)

Before #233 — the lying green
✓ green

No auth secret → the review never ran, but continue-on-error swallowed the failure. The only trace was a ::warning:: buried inside a green check. Green looked like "reviewed" when nothing was reviewed.

Now — the check outcome IS the signal
GREEN ✓ Review actually ran AND succeeded.
RED ✗ Review did NOT run — no auth secret, bad token, plan lockout, or action outage.

RED ≠ "found problems".
RED = "not reviewed" — and advisory, so it never blocks the merge.

No continue-on-error mask anywhere in either lane; a ::warning:: inside a green check was judged not enough.

04

Auth resolver & activation

Resolver priority — identical in both lanes

1
CLAUDE_CODE_OAUTH_TOKEN preferred
Subscription OAuth (Pro/Max). Mint on any machine already logged in to Claude: 
claude setup-token   # prints a CLAUDE_CODE_OAUTH_TOKEN
2
ANTHROPIC_API_KEY fallback
API-key billing — an isolated dollar budget instead of the subscription.
3
Neither present RED — exit 1
Visible skip: ::error annotation + executed=false step summary. Never a silent green.

Secrets are referenced by NAME only — never a value in code, commits, issues, or chat.

Activate (one-time, owner)

Org-wide covers every current + future repo in one shot (recommended):

# org-wide — this project's org is 3D-Stories
gh secret set CLAUDE_CODE_OAUTH_TOKEN --org 3D-Stories --visibility all

# or per-repo, if you don't own the org
gh secret set CLAUDE_CODE_OAUTH_TOKEN --repo <owner>/<repo>

# API-key fallback variant
gh secret set ANTHROPIC_API_KEY --org <your-org> --visibility all

gh prompts for the value — it is never echoed. Either secret activates both lanes.

Zero-secret alternative: run the lanes on a self-hosted runner already logged in to Claude — no repo secret needed. See docs/config-reference.md#ci-review-auth.

SECRET ALONE IS NOT ENOUGH. The Claude Code GitHub App (github.com/apps/claude) must also be installed on the repo/org, or the action fails with 401 … not installed on this repository.
05

What a run produces

① Inline PR comments — the findings

Both lanes post Claude's confirmed findings as inline PR review comments on the diff (security lane via the action's classify_inline_comments buffering). This is where the review substance lands — the check status only says whether a review happened.

② PR check status

One check per lane: green = reviewed or red = not reviewed. Both advisory — not required checks; red never blocks the merge (until owner opts in, §06).

③ GITHUB_STEP_SUMMARY — exact formats

Success (auth mode is oauth or apikey):

## Claude code review: executed=true (auth=oauth) — reviewer_kind=builtin_code_review
## Claude security review: executed=true (auth=oauth)

No auth secret (both lanes, RED):

## Claude <code|security> review: executed=false (no auth secret) — check is
RED to signal 'not reviewed', never green. Advisory / non-blocking.
Activate: docs/ci-review-lanes.md

Auth present but review step failed:

## Claude code review: executed=false (review step did not succeed —
does NOT count toward the #162 A/B sample)
## Claude security review: executed=false (review step did not succeed —
does NOT count toward the 10-PR promotion tally)

④ Telemetry — the #162 A/B experiment

The code-review lane is the builtin_code_review arm of the #162 data gate: built-in /code-review vs WF2's hand_rolled_multi Step 11 panel, compared on findings-yield-per-token over ≥10 runs. The Action reports token usage per run; each review is recorded as a run-record gate entry with the usage block (input/output tokens, cost) — see docs/run-records.md#builtin-code-review-capture. A run counts toward the sample only if it executed — failed auth or a failed review step is excluded by the record-status step itself. WF5's Codex cross-model diff pass is unchanged.

06

Verify · triage · promote

VERIFY a run happened

Open the lane's Action run → Summary. A working lane prints:

Claude <security|code> review:
executed=true (auth=oauth)

Still red with executed=false (no auth secret)? The secret isn't visible to the workflow — wrong org/repo scope, or the App isn't installed.

TRIAGE a red lane

Red is a triage question, never an automatic blocker:

  • Infra red — no secret / bad token / lockout / outage. Nothing was reviewed; fix auth, or rely on WF2's Step 11 / 11.5 which already gated the PR.
  • Ran-but-failed — read the run log; the ::warning marks it "not a clean signal".

Findings themselves arrive as PR comments — a lane with findings is still green.

PROMOTE to a hard gate (opt-in)

To make a lane block merges: mark it a required status check in branch protection. Recommended only after ~10 clean (executed=true) PRs, per the lanes' own promotion note.

A required red then blocks exactly as intended — the honest signal makes the promotion safe.
