# Rawgentic Plugin Overhaul: Design Document

**Date:** 2026-03-06
**Status:** Approved (post-critique revision)
**Brainstorming session:** docs/brainstorming/session_20260306_113924.md
**Critique report:** docs/plans/2026-03-06-plugin-overhaul-critique.md

---

## Problem Statement

The rawgentic plugin was built assuming a React/Node.js project deployed on Docker. Skills contain hardcoded references to npm, Docker Compose, Express.js, PostgreSQL, and specific file patterns. There is no concept of multi-project workspaces, no structured config file, and no mechanism for switching between projects.

## Goals

1. Make all skills project-agnostic and technology-agnostic
2. Move project configuration from CLAUDE.md to a structured `.rawgentic.json` per project
3. Support multi-project workspaces with project switching
4. Support both existing projects (auto-detect) and new projects (brainstorm intent)
5. Create a "learning config" pattern where skills update config when they discover new capabilities

## Non-Goals

- Project scaffolding (folder structure, package init) — rawgentic does SDLC workflows, not bootstrapping
- Supporting non-GitHub VCS providers (GitHub-first, extensible later)
- Changing the 14 principles (P1-P14) or step archetypes (A-L)

## Architecture Overview

```
Claude Root (e.g., ~/claude/)
|-- .rawgentic_workspace.json        # Project registry (which projects, which is active)
|-- CLAUDE.md                        # Static pointer: "Rawgentic workspace: .rawgentic_workspace.json"
|-- projects/
|   |-- my-app/
|   |   |-- .rawgentic.json          # Deep project config (tech stack, services, testing, etc.)
|   |   |-- src/
|   |   |-- ...
|   |-- rawgentic/
|   |   |-- .rawgentic.json
|   |   |-- skills/
|   |   |-- ...
|   |-- other-project/
|       |-- .rawgentic.json
|       |-- ...
```

**Two layers:**
- **Workspace layer** — `.rawgentic_workspace.json` + `new-project` + `switch` skills — manages WHICH project
- **Config layer** — `.rawgentic.json` + `setup` skill — manages HOW a project is configured

**Runtime flow:**
1. Session-start hook reads workspace file, injects active project context
2. User invokes a skill (e.g., `/rawgentic:implement-feature 42`)
3. Skill's `<config-loading>` block reads workspace -> finds active project -> reads `.rawgentic.json`
4. Skill executes using `config` and derived `capabilities` — no filesystem probing for config data
5. If skill discovers new capabilities, it updates `.rawgentic.json` following the merge policy before completing

## User Journeys

### Cold Start (first-time user)
1. User installs rawgentic plugin
2. Session-start hook: "No rawgentic workspace found. Run /rawgentic:new-project to get started."
3. User runs `/rawgentic:new-project my-app`
4. `new-project` creates `./projects/my-app/`, asks about GitHub clone vs brand new
5. `new-project` creates `.rawgentic_workspace.json`, registers project, delegates to `setup`
6. `setup` auto-detects tech stack (or brainstorms for blank project), writes `.rawgentic.json`
7. User can now invoke any workflow skill

### Returning User
1. Session-start hook: "Active project: my-app (./projects/my-app)."
2. User invokes a workflow skill directly — config-loading reads active project's `.rawgentic.json`

### Switching Projects
1. User says "change project to rawgentic" or runs `/rawgentic:switch rawgentic`
2. `switch` updates workspace: rawgentic becomes active, my-app deactivated
3. All subsequent skill invocations use rawgentic's `.rawgentic.json`

### Migrating from Old Rawgentic
1. User has existing CLAUDE.md with "## Project Constants (generated by /rawgentic:setup)"
2. Runs `/rawgentic:new-project` to register the project
3. `setup` detects old CLAUDE.md section, parses existing constants as seed values
4. Writes `.rawgentic.json`, removes old SDLC section from CLAUDE.md, adds static pointer
5. Informs user: "Migrated project constants from CLAUDE.md to .rawgentic.json"

---

## CLAUDE.md Content

When rawgentic is configured, CLAUDE.md should contain only a static pointer:

```markdown
## Rawgentic
Workspace config: .rawgentic_workspace.json
```

This is written by `setup` (on first configuration) or `new-project` (on workspace creation) if not already present. The pointer never changes — it points to the workspace file, which in turn tracks the active project.

**Migration:** If CLAUDE.md contains the old `## SDLC Workflow Principles` or `## Project Constants (generated by /rawgentic:setup)` sections, `setup` removes them and replaces with the pointer block.

## Schema: `.rawgentic_workspace.json`

Location: Claude root directory.

```json
{
  "version": 1,
  "projectsDir": "./projects",
  "projects": [
    {
      "name": "string",
      "path": "string (relative to Claude root)",
      "active": "boolean (exactly one true if projects exist)",
      "lastUsed": "ISO 8601 timestamp",
      "configured": "boolean (true after setup completes)"
    }
  ]
}
```

**Invariants:**
- If `projects.length > 0`, exactly one has `active: true`
- `new-project` adds entry with `active: true` (deactivates previous)
- `switch` sets target `active: true` (deactivates previous), updates `lastUsed`
- `setup` sets `configured: true` on completion
- `lastUsed` updated by `switch` and `new-project` only (NOT by every skill invocation)

**Path handling:** Projects registered outside `projectsDir` are stored with their specified path (relative or absolute). `projectsDir` is used only as the default base when `new-project` receives a bare name.

## Schema: `.rawgentic.json`

Location: Project root directory (e.g., `./projects/my-app/.rawgentic.json`).

```json
{
  "version": 1,
  "project": {
    "name": "string (required)",
    "type": "application | infrastructure | scripts | library | docs | research (required)",
    "description": "string (required)"
  },
  "repo": {
    "provider": "github (required)",
    "fullName": "org/repo (required)",
    "defaultBranch": "main (required)"
  },
  "techStack": ["string (optional, informational only)"],
  "testing": {
    "frameworks": [
      {
        "name": "string",
        "type": "unit | e2e | integration",
        "command": "string",
        "configFile": "string (optional)",
        "testDir": "string (optional)"
      }
    ]
  },
  "database": {
    "type": "string (required if section present)",
    "cli": "string (required if section present)",
    "name": "string (optional)",
    "user": "string (optional)",
    "container": "string (optional)",
    "migrationsDir": "string (optional)",
    "migrationPattern": "string (optional)"
  },
  "services": [
    {
      "name": "string",
      "type": "string (e.g., frontend, backend, worker, api, proxy, scheduler)",
      "framework": "string",
      "port": "number",
      "entryPoint": "string (optional)",
      "healthCheck": "string (optional)"
    }
  ],
  "infrastructure": {
    "hosts": [
      {
        "name": "string",
        "address": "string",
        "sshUser": "string",
        "description": "string (optional)"
      }
    ],
    "docker": {
      "composeFiles": [
        { "name": "string", "path": "string" }
      ]
    }
  },
  "deploy": {
    "method": "compose | script | ssh | manual",
    "command": "string (optional)",
    "targetHost": "string (optional)"
  },
  "security": {
    "authMechanisms": [
      {
        "type": "string",
        "appliesTo": ["string (service names)"],
        "middlewarePath": "string (optional)",
        "headerName": "string (optional)"
      }
    ],
    "validationLibrary": "string (optional)",
    "dataChannels": ["string"]
  },
  "ci": {
    "provider": "string (optional section)",
    "workflowDir": "string"
  },
  "formatting": {
    "tool": "string (optional section)",
    "configFile": "string",
    "command": "string"
  },
  "documentation": {
    "primaryFiles": ["string"],
    "format": "string"
  },
  "custom": {}
}
```

**Required fields:** `version`, `project` (name, type, description), `repo` (provider, fullName, defaultBranch)
**All other sections:** Optional. Skills adapt behavior based on what's present. Missing section = capability not available.

**Key design choices (post-critique):**
- `version` field enables schema migration. Config-loading protocol checks version against expected.
- `techStack` is optional and informational only. All capability decisions derive from structured sections (testing, database, services, etc.), not from string matching on techStack.
- `project.root` removed — project path is already in `.rawgentic_workspace.json`.
- `services[].type` is a free-form string, not a closed enum — supports any service type.
- `database.type` and `database.cli` are required IF the database section is present.
- `custom` section provides an escape hatch for project-specific constants not covered by the schema.

## Session-start Hook

Bash script. Reads `.rawgentic_workspace.json`, injects context via `hookSpecificOutput.additionalContext`.

**Working directory handling:** The hook uses the Claude project root (the directory where Claude Code was launched). If `.rawgentic_workspace.json` is not found in the current directory, the hook checks the parent directory (to handle cases where Claude is launched from a project subdirectory).

**3-state flow:**

| State | Condition | Injected Context |
|-------|-----------|------------------|
| No workspace | `.rawgentic_workspace.json` missing | "No rawgentic workspace found. Run /rawgentic:new-project to get started." |
| No projects | Workspace exists, `projects` empty | "Rawgentic workspace exists but no projects registered. Run /rawgentic:new-project." |
| Active project | Active project found | "Active project: <name> (<path>)." + "Config missing -- run /rawgentic:setup." if not configured |

**Error handling:** If `.rawgentic_workspace.json` exists but is malformed JSON, inject: "Rawgentic workspace file is corrupted. Run /rawgentic:new-project to regenerate, or fix .rawgentic_workspace.json manually."

No interactive prompting. No project switching. No reading `.rawgentic.json`. Pure context injection.

## Config-loading Protocol

Inline `<config-loading>` block included at the top of every workflow skill (identical in each):

```
<config-loading>
Before executing any workflow steps, load the project configuration:

1. Read `.rawgentic_workspace.json` from the Claude root directory.
   - Missing -> STOP. Tell user: "No rawgentic workspace found. Run /rawgentic:new-project."
   - Malformed JSON -> STOP. Tell user: "Workspace file is corrupted at <path>.
     Run /rawgentic:new-project to regenerate, or fix manually."
   - Extract the active project entry (active == true).

2. Read `<activeProject.path>/.rawgentic.json`.
   - Missing -> STOP. Tell user: "Active project <name> has no config. Run /rawgentic:setup."
   - Malformed JSON -> STOP. Tell user: "Project config is corrupted at <path>.
     Run /rawgentic:setup to regenerate."
   - Check `config.version`. If version > 1 (or missing), warn: "Config file uses
     schema version <N>, but this skill expects version 1. Run /rawgentic:setup --upgrade
     or update the rawgentic plugin."
   - Parse full JSON into `config` object.

3. Build the `capabilities` object from config:
   - has_tests: config.testing exists AND config.testing.frameworks.length > 0
   - test_commands: config.testing.frameworks[].command
   - has_ci: config.ci exists AND config.ci.provider exists
   - has_deploy: config.deploy exists AND config.deploy.method exists and != "manual"
   - has_database: config.database exists AND config.database.type exists
   - has_docker: config.infrastructure exists AND config.infrastructure.docker.composeFiles.length > 0
   - project_type: config.project.type

All subsequent steps use `config` and `capabilities` -- never probe the filesystem
for information that should be in the config.

If a skill discovers new project capabilities during execution (e.g., a new test
framework, a previously unknown service), update `.rawgentic.json` following the
Learning Config Merge Policy before completing the workflow.
</config-loading>
```

## Learning Config Merge Policy

When a skill discovers new project capabilities during execution:

**Allowed operations:**
- **Append** to arrays — e.g., add a newly discovered test framework to `testing.frameworks[]`
- **Set** fields that are currently `null`, missing, or empty — e.g., populate `database.migrationsDir` if discovered
- **Add** new optional sections that didn't exist — e.g., add `ci` section if `.github/workflows/` found

**Not allowed without user confirmation:**
- **Overwrite** existing non-null values — if config says `database.type: "postgres"` and skill thinks it's MySQL, ASK the user
- **Remove** any existing entries from arrays or sections
- **Change** required fields (`project.name`, `repo.fullName`, etc.)

**Conflict resolution:** If a skill discovers a value that conflicts with an existing config value, log the discrepancy in session notes and present it to the user at the end of the skill execution: "Config update: I found X but your config says Y. Would you like to update?"

**Write safety:** Always read the full file, modify in memory, write the full file back. Never do partial patches.

## Skill Designs

### `rawgentic:setup` (rewrite)

**Purpose:** Configure `.rawgentic.json` for the active project.

**8 steps:**
1. **Verify context** — Read workspace, find active project. STOP if none.
2. **Migration check** — If CLAUDE.md contains `## Project Constants (generated by /rawgentic:setup)` or `## SDLC Workflow Principles`:
   - Parse existing constants as seed values for detection
   - Flag for cleanup in step 7
3. **Detect or brainstorm:**
   - **Has existing `.rawgentic.json`** -> Read it. Present current config. Ask user what to update. Merge changes preserving learned entries from previous skill runs.
   - **Has code but no config** -> Auto-detect: package.json, pyproject.toml, go.mod, Cargo.toml, Dockerfile, docker-compose, .github/workflows, .env files, git remote, auth patterns, validation libraries, services. Use migration seed values if available.
   - **Empty/new project** -> `/superpowers:brainstorm` for project intent, planned tech, type classification
4. **Present detected config** — Show filled `.rawgentic.json` with detection sources
5. **User confirms/edits** — Section by section, only show relevant sections
6. **Write `.rawgentic.json`** — To `<activeProject.path>/.rawgentic.json` with `"version": 1`
7. **Update CLAUDE.md** — If old SDLC/constants sections exist, remove them. Ensure static pointer block is present.
8. **Update workspace** — Set `configured: true`
9. **Verify** — File written, required fields present, JSON valid, repo accessible via `gh repo view`

### `rawgentic:new-project` (new)

**Purpose:** Register a project in workspace and delegate to setup.

**Invocation:** `/rawgentic:new-project my-app` or `/rawgentic:new-project ./projects/my-app`

**6 steps:**
1. **Parse input** — Name or path. Derive name from last path segment, or construct path as `<projectsDir>/<name>`.
2. **Check if registered** — Look in `.rawgentic_workspace.json` by name and path. If found, offer switch or re-setup.
3. **Folder check:**
   - **Doesn't exist** -> Create `<path>/`. Ask: "Is there a GitHub repo to clone?"
     - **Yes** -> Get repo URL. `git clone <url> <path>`. If clone fails, remove created folder and STOP with error. Do NOT register a failed clone.
     - **No** -> `git init <path>`.
   - **Exists** -> Verify directory is accessible. Proceed.
4. **Create workspace file if missing** — If `.rawgentic_workspace.json` does not exist, create it with `{"version": 1, "projectsDir": "./projects", "projects": []}`.
5. **Register** — Add project entry with `active: true`, `configured: false`, `lastUsed: now()`. Deactivate any previously active project.
6. **Delegate** — Invoke `rawgentic:setup` for the newly active project.

### `rawgentic:switch` (new)

**Purpose:** Change the active project.

**Invocation styles:**
- `/rawgentic:switch my-app`
- `/rawgentic:switch ./projects/my-app`
- "change project to my-app"
- "change project to ./projects/my-app"

**5 steps:**
1. **Parse input** — Name or path.
2. **Find in workspace** — Match by `name` first, then by `path`. If not found, suggest `/rawgentic:new-project`.
3. **Verify directory exists** — Check that the project's directory is still present on disk. If deleted, warn user and offer to unregister or re-create.
4. **Switch** — Set target `active: true`, previous `active: false`, update `lastUsed` on target.
5. **Confirm** — Report:
   ```
   Switched to: <name> (<path>)
   Configured: yes/no
   Last used: <date>
   ```
   If not configured, suggest running `/rawgentic:setup`.

## Phase 2-3: Workflow Skill Updates (deferred)

After Phase 1 foundation is complete, all 9 workflow skills need:
1. Add `<config-loading>` block at top (with version check and corruption handling)
2. Remove all CLAUDE.md constant references
3. Replace hardcoded tech-stack commands with config-driven equivalents
4. Add "learning config" instructions following the merge policy
5. Remove `<capability-detection>` sections that probe filesystem for config-level data
6. Add handling for `research` project type where applicable

Priority order (by hardcoding severity):
1. implement-feature (reference pattern)
2. fix-bug (heavy infra assumptions)
3. incident (heavy infra assumptions)
4. security-audit (framework-specific)
5. optimize-perf (profiling tools)
6. update-deps (package managers)
7. refactor (light touch)
8. update-docs (light touch)
9. create-issue (mostly clean)

Approach for Phase 2-3 to be decided after Phase 1 completion.

## Known Limitations (to address post-Phase 1)

- **No `unregister`/`remove` skill** — workspace accumulates stale entries. Add later.
- **Single deploy target** — `deploy` section supports one method/command. Multi-target (dev/staging/prod) deferred.
- **No general `scripts`/`commands` section** — use `custom: {}` as escape hatch for now.
- **Config-loading protocol is duplicated** — inline in each skill per design decision. If drift becomes a problem, add a validation script to check consistency across skills.
