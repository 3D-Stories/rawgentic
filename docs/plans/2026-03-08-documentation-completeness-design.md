# Documentation Completeness — Design

**Date:** 2026-03-08
**Category:** addition (with minor corrections)
**Status:** Approved

## Problem

The rawgentic docs/ directory has comprehensive coverage for workflows (9 design specs), principles (P1-P14), and the unified framework (consolidation.md). However, a completeness audit against the codebase found 5 gaps where implemented functionality has no contributor-facing documentation. Additionally, skill counts in README.md and plugin.json were inaccurate.

## Decisions Made

- **Audience:** Future contributors (onboarding-level explanations)
- **Location:** Flat in `docs/` alongside existing files (no new subdirectory)
- **Config reference:** Hybrid — short principles doc + inline `$comment` fields in the JSON schema
- **Testing doc:** Documents what exists + placeholder sections for known gaps, linking to issue #9
- **Approach:** Minimal reference docs — each file answers one question, no overlap

## New Documents

### 1. `docs/testing.md` (~70 lines)

**Question it answers:** "How do I run the tests, and what's tested vs. not?"

Sections:
- **Running Tests** — Prerequisites (Python 3, pytest), command (`pytest tests/`)
- **Security Guard Tests**
  - Unit tests (`tests/hooks/test_security_guard.py`) — covers glob_match, normalize_path, extract_content, check_word_boundary, sanitize_path_for_message, suggest_glob, match_patterns, filter_exceptions, format_deny
  - E2E tests (`tests/hooks/test_security_guard_e2e.py`) — end-to-end pattern blocking
- **Tests to be Implemented**
  - WAL hooks (wal-pre/post/fail/stop) — JSONL entry correctness
  - wal-context — context injection scenarios
  - wal-bind-guard — cross-project write blocking
  - wal-guard — dangerous command detection
  - session-start — WAL recovery, rotation, reconciliation, archival
  - security-guard-check — conflict detection
  - Links to issue #9

### 2. `docs/wal-guide.md` (~90 lines)

**Question it answers:** "How does WAL work and how do I inspect/debug it?"

Sections:
- **Overview** — What WAL does, per-project files (`claude_docs/wal/<project>.jsonl`), legacy fallback
- **WAL Entry Format** — JSON schema of a single entry, example entries per phase
- **WAL Lifecycle** — wal-pre (INTENT), wal-post (DONE), wal-post-fail (FAIL), wal-stop (SESSION_END)
- **Inspecting WAL Files** — Common jq queries (incomplete ops, filter by session/tool, count ops)
- **WAL Recovery** — session-start detection, rotation (>5000 lines, >7 days), sanitization
- **Troubleshooting** — Empty WAL, corrupt entries, missing WAL file

### 3. `docs/skill-development.md` (~80 lines)

**Question it answers:** "How are skills structured, and what are the evaluation artifacts?"

Sections:
- **Skill Structure** — `skills/<name>/SKILL.md`, frontmatter (name, description), body as prompt
- **Workspace Directories** — `skills/<name>-workspace/`, created during evaluation not runtime, structure (iteration-1/benchmark.md, scenarios with with_skill/without_skill transcripts)
- **Evaluation Methodology** — Scenario testing, with/without comparison, benchmark grading, phase2-eval-summary.md
- **Adding a New Skill** — Create SKILL.md, register in plugin.json, optional evaluation workspace

### 4. `docs/session-notes.md` (~65 lines)

**Question it answers:** "How does the session notes system work?"

Sections:
- **Overview** — Per-project markdown files, location (`claude_docs/session_notes/<project>.md`), auto-creation
- **How Notes are Populated** — session-start initialization, wal-context reads status, workflow step markers (`### WF<N> Step X: <Name> — DONE`), context-compaction recovery
- **Session Registry** — `claude_docs/session_registry.jsonl`, entry format, written by /rawgentic:switch or auto-bind, read by wal-context
- **Session Lifecycle** — 7-step flow from start to archival
- **Archival** — Stale notes preserved on next startup

### 5. `docs/config-reference.md` (~65 lines) + schema updates

**Question it answers:** "What are the config design principles?"

Part A — `docs/config-reference.md`:
- **Overview** — .rawgentic.json purpose, generated by /rawgentic:setup, link to schema
- **Design Principles** — Config-driven, single source of truth, capabilities object, learning config
- **Core Sections** — Brief description of each top-level key (project, repo, techStack, testing, database, services, infrastructure, deploy, security, ci, formatting, documentation, custom)
- **Versioning** — version = 1, mismatch warnings
- **Config-Loading Protocol** — Fallback chain, path resolution, missing config behavior

Part B — `templates/rawgentic-json-schema.json`:
- Add `$comment` fields to each section explaining purpose and which workflows consume it
