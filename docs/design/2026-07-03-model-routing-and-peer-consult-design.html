Problem

Every subagent inherits the session model; Codex can only tear down, never propose

Rawgentic workflow skills dispatch subagents at seven sites — none specifies a model. On a premium session model (Fable), the review fleets (3 judges + 2 per-task reviewers + 3 pre-PR reviewers per WF2 run) burn premium tokens on work a cheaper strong model handles well. Separately, Codex participates only as WF5's adversarial reviewer; there is no way to engage it as a peer designer — same problem, blind, its own proposal.

Rejected alternative: ad-hoc orchestration (a standing "you are the orchestrator" prompt + registered agents). Dispatch knowledge would live in per-run improvisation instead of versioned skill text; agent definitions load only at session start (fragile); WF2 deliberately uses inline prompt roles ("NOT registered subagents", Step 8a).

Feature 1

modelRouting — role→model config, default-off

Optional block in the project's .rawgentic_workspace.json entry (same placement as adversarialReview, critiqueMethod, headlessEnabled):

"modelRouting": { "review": "opus", "analysis": "sonnet", "implementation": "opus" }
  • Roles: review (all judge/reviewer fleets), analysis (Step 2 gather + Step 10 memorization), implementation (Step 8 task delegation — below).
  • Values: opus · sonnet · haiku · fable · inherit. Partial config valid.
  • Default: absent block/role or inherit → session model. Byte-identical to v2.45.0 unless configured.

Setup integration

/rawgentic:setup gains a modelRouting step (three roles, suggested defaults, skip-per-role = inherit) and a peerConsult enable step mirroring the adversarialReview one — both applied in the same single finalize read-modify-write as the other per-project fields.

Resolution library

python3 hooks/model_routing_lib.py resolve \
  --workspace .rawgentic_workspace.json --project <name> --role review

stdout = model or inherit, exit 0 in all degradable cases. Unknown value, malformed block, missing project → stderr warning + inherit. Fail-open by design: routing is an optimization knob, not a security gate — nothing here may block a workflow run. Soft floor: review below opus resolves as configured but warns ("below recommended opus floor") — explicit sonnet/haiku only; inherit and fable never warn.

Skill changes — 3 skills, 7 sites

Config-loading preambles of implement-feature, fix-bug, refactor gain one resolve call per used role, carried as literals (fresh-shell rule). Each Agent dispatch adds model: unless inherit. Codex/adversarial dispatches untouched. No registered agents.

Step 8 implementation delegation (implementation role)

When configured, WF2 Step 8 delegates each plan task to a subagent instead of executing inline:

  • Brief per task: design doc + plan task + TDD requirement + conventions + current test baseline.
  • Serial: one task-agent at a time; each task builds on the previous commit. No parallel dispatch in v1.
  • Main loop stays orchestrator: re-runs the suite after each task, diffs against baseline, proceeds or intervenes. Step 8a reviews unchanged.
  • Fail-open with a clean-state boundary: pre-task state recorded (HEAD + porcelain status); dead/vacuous task-agent → restore pre-task state first, then retry once inline, log, continue. Successful tasks end committed. Delegation can never block Step 8; inline retry never runs on a half-mutated tree.
  • Why safe: the subagent gets a validated plan task (post Step 4 critique + Step 6 drift check), not an open problem. Proven 2026-07-03: #123 executed to a green PR by an Opus agent from a main-loop brief.
  • Absent role → inline, exactly today's behavior.

Feature 2

peerConsult — WF13: Codex as peer designer, standalone + WF2 integration

The WF5 pattern: a standalone skill /rawgentic:peer-consult registered as WF13 (WF6 stays reserved per docs/consolidation.md; highest used was WF12), plus opt-in WF2 Step 3 wiring — exactly how WF5 pairs /rawgentic:adversarial-review with its gate integration. Standalone: takes a problem artifact, writes the peer proposal to docs/reviews/peer-<slug>-<date>.md, report-only, same egress conventions as WF5; always invocable regardless of config.

"peerConsult": { "enabled": true, "workflows": ["implement-feature"] }

Same shape and placement as adversarialReview; default-off; governs only the WF2 integration; v1 wires only implement-feature. Framing (verbatim into the prompt file): a peer senior engineer — on par with the reasoning tier, a different perspective; a peer, not a reviewer.

1

Blind both ways — mechanism. WF2 launches the WF13 engine as a background codex exec writing structured output to a temp file (-o); prompt = issue body + Step 2 analysis summary (the same context Claude designs from), inlined per the codex sandbox read restriction. The main loop records the output path but must not read it until Claude's own draft is written to the design-doc file. Completion = process exit; timeout/failure writes an explicit empty-proposal marker, never partial content. Claude + Codex concurrency is safe — separate quota pools.

2

Synthesis. After Claude's draft is on disk, read the peer proposal, merge best-of-both; the design doc records the peer's contributions (provenance).

3

Gates unchanged. Step 4 judges critique the synthesized design. WF5 keeps its adversarial stance and #121 tuning — peer consult is generative (design phase), adversarial review is evaluative (gate phase); both may be on.

4

Non-blocking. Codex failure → log, proceed with Claude's design alone (same posture as WF5-in-WF2).

Plumbing: extend adversarial_review_lib.py with a consult mode reusing the shared codex invocation/prereq/egress code (different prompt + output schema: a proposal, not findings); is-enabled gains --key (default adversarialReview = backward compatible; new peerConsult). One lib, no parallel copy. Prompt in skills/peer-consult/ (the WF13 skill's own directory). docs/consolidation.md + README workflow table register WF13.

Error handling

Single failure point, always degrades open

model_routing_lib.py is the only routing failure point and always degrades to inherit with a warning; skills never parse workspace JSON themselves for routing. Missing lib (stale plugin cache) → treated as inherit. Peer-consult failure never blocks Step 3. Unknown config fields are ignored, never errored.

Testing

Unit tests + drift guards

  • tests/hooks/test_model_routing.py: absent block / partial roles / invalid value / floor warning / malformed JSON / missing project / inherit passthrough.
  • adversarial_review_lib tests extended: --key default compatibility + peerConsult.
  • Skill-lint: each of the 7 dispatch sites must carry its role annotation (a new dispatch site cannot silently bypass routing); peer prompt file exists; Step 3 sub-step documents the non-blocking path; Step 8 delegation documents the brief template and retry-once-inline fallback.
  • model_routing_lib tests cover implementation identically to the other roles.
  • WF13 registration test (mirrors the WF5 one); consult-mode lib tests (schema, empty-proposal marker on timeout); setup-step lint for the new prompts.
  • Acceptance criterion (future, verified at implementation): full suite green against the recorded pre-implementation baseline — expected 1384 passed / 5 warnings as of spec time (post-#123).
Release

Docs + version

docs/config-reference.md gains both sections; README operational sections updated (changelog history stays verbatim per the #123 AC8 amendment); version 2.45.x → 2.46.0, one PR for both features. Marketplace manifest carries no plugin version (verified in #123) — nothing to sync. Implementation proceeds from this spec via an implementation plan; tracking issue filed at planning time.

Out of scope

Named and left

  • Registered/plugin-shipped agents (contradicts WF2's inline-role design; session-load fragility).
  • Parallel task dispatch within Step 8 delegation (serial only in v1).
  • Effort-level routing (model only in v1).
  • Peer consult wired into fix-bug/refactor (config-ready only).
  • Any change to WF5's adversarial stance or #121 tuning.
