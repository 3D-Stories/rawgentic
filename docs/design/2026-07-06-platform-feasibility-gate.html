Platform / external-dependency feasibility gate (#226)

WF2 Step 3 design of record · Complexity: standard_feature · Blast: low code, prose-heavy · plugin 3.12.0

Design shaped by a blind cross-model peer consult (converged on Approach A) and a cross-model adversarial-on-design pass (2 High folded → the mechanical omission-catch). The pre-PR review round (a cross-model diff review + two reviewer agents) then closed five more fail-opens — chiefly docs removed from accepted evidence kinds and duplicate declarations rejected as ambiguous. Dogfooded: this feature's own design declares platform_apis: none and passes its own Step-4 gate.

Problem

A design can commit to a platform/framework API the project's own configuration does not permit and still pass every current gate, because the gates are test-centric and the failure only manifests at runtime on a surface CI never exercises. Motivating real run: a Tauri 2 overlay used window.setSize; that window's capability file lacked allow-set-size (the main window had it — a visible asymmetry in the repo). The call was silently denied (error only in the webview console); cargo compile, tsc, and unit tests were all green. Three build-and-UAT cycles to catch.

Unifying principle: an unproven external/platform dependency is a design-gate blocker, verified against the project's real config — not assumed from the API's existence.

Approaches weighed

  • chosen A. Evidence-cited feasibility declaration + review lens + tiny validator. A design declares a mandatory platform_apis: block cited to real evidence; the design gate (Step 4 self-review + the WF5 adversarial lens) checks it; a small plan_lib helper mechanically rejects an omitted declaration, an assumed dep, weak evidence, or an unsurfaced silent failure; drift guards pin the prose. Generic and framework-agnostic.
  • rejected B. Automated capability-model parser (parse Tauri capability files, iOS entitlements, browser permission policies, …). Explicitly out-of-scope in the issue: per-framework, unbounded to maintain, and the issue wants a generic, evidence-cited mechanism, not an auto-parser. Evidence is a cited file/doc/call-site/spike, not a parse.

The canonical feasibility contract

A platform_apis: declaration is mandatory on every design doc / RCA — the key decision resolving the adversarial High findings: an omitted note must not pass silently, which is the exact silent-gap class #226 targets. It is one line when there are no platform APIs — symmetric with the already-required "Security implications" section, so it is not the "feasibility proof for APIs that are fine" over-gating the issue warns against.

## Platform / external dependencies
platform_apis: none        # the whole declaration when no material platform/external API is used

or, per material platform/framework/external API not already proven in-repo the same way, one block per API:

platform_apis:
- api: <exact API> on <exact object/runtime surface>
  feasibility: verified via <capabilities-file|existing-call-site|spike> — <citation>
  failure: fail-loud | fail-silent
  surface: <assertion|log|observable check> — <where>   # REQUIRED when failure: fail-silent
  • feasibility: assumed may appear as an interim Step-3 drafting marker, but is explicitly Step-4-blocking.
  • docs is NOT an accepted evidence kind: docs prove the API exists, not that this project's config permits it — accepting docs is the exact #226 failure (docs say setSize exists; the capability file denies it). Cite a capabilities/manifest file, an exact call site, or a spike. (A cross-model diff review flagged docs-accepted as a fail-open; closed mechanically.)
  • Working-precedent: an existing-call-site counts only for the exact API on the exact object kind + target surface (the main-window-has-it / overlay-doesn't asymmetry); otherwise spike or cite config.
  • The full contract lives once in WF2 §3; WF3 and the lens carry a short checklist + pointer, not a duplicate.

Mechanism

1. plan_lib feasibility validator (the only new code)

  • parse_feasibility_block(text) → FeasibilityDecl | None — returns None when NO platform_apis: declaration is present (so the caller can distinguish "declaration absent" from "declared none"); code-fence aware (both ``` and ~~~); collects ALL non-fenced declarations — >1 sets ambiguous=True so an early stray none can't shadow a later real one. Separator-tolerant (em/en-dash, minus, hyphen, colon).
  • assert_feasibility_declared(decl) → (ok, errors) — the mechanical Step-4 gate, fail-closed: None (absent) errors; ambiguous (>1 declaration) errors; none passes; else per block assumed, a bad evidence-kind (docs excluded), an empty citation, a missing failure:, or fail-silent without surface: all error.

No new dataclass field is needed for the runtime-surface rule — the existing deferral_reason and the run-record's verification_deferred[].target_check already carry "the claim most likely to be wrong." A mechanical "is this the most likely wrong claim?" check is impossible; the drift guard pins the requirement and review judges the naming.

Failure modes / over-gating traps (the main risk)

  • Over-gating (the issue's stated #1 risk): the feasibility proof requirement is scoped to APIs not already proven in-repo the same way — a precedented exact call site is sufficient, so well-worn APIs never trip it. The mandatory declaration is one line (platform_apis: none), symmetric with the existing "Security implications" section — not proof-for-everything.
  • False "verified": the validator requires an allowed evidence kind (docs excluded) AND a non-empty citation, so verified via docs, an unknown kind, or a citation-less verified all fail closed.
  • Silent skip of the mirror: the drift guard pins the WF3 §3/§4 text (and every other surface) so a later prose edit can't quietly drop it.

Security / backward-compat

  • Additive: the validator returns None/ok on absent notes only via the explicit gate decision; no change to parse_tasks, the deferral machinery, or any existing contract. No new dependency, no migration. Pure-function helpers, unit-tested.
