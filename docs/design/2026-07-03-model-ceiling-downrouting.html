modelRouting.implementation as a Ceiling

Target v2.47.1 Basis feature/132-model-ceiling-downrouting @ a7059a5 Companion same-name .md in docs/design/
Problem

A blanket assignment overspends on well-specified work

WF2 Step 8's delegation dispatched every plan task with model: <implementation>. With implementation: opus, well-specified, 2-file, mechanical/TDD tasks that Sonnet already handles fine were still built on Opus — over-spend the project owner flagged directly. The signals to do better already exist and were sitting unused: Step 2's complexity classification (simple_change / standard_feature / complex_feature) and Step 5's per-task riskLevel: high | standard (plan_lib.RISK_CRITERIA).

Approach: stop treating modelRouting.implementation as a floor every task inherits, and start treating it as a ceiling — the most expensive model a task is allowed to reach, not the one it defaults to.
Approach

select_impl_model(ceiling, riskLevel, complexity) → (model, reason)

One pure function added to hooks/model_routing_lib.py. Fail-open, never raises. Returns a (model, reason) pair — the reason is logged per task for audit (AC3).

ModelRankRole in the ceiling clamp

haiku—deliberately unrankable for implementation — standing project rule: never Haiku for coding
1

ceiling == "inherit" → inherit, "no routing configured — session model." Routing off ⇒ no per-task logic; Step 8 behaves exactly as before this feature.

2

ceiling not one of {sonnet, opus, fable} (i.e. haiku or unknown) → floors to sonnet, "not a valid implementation model — floor to sonnet." A misconfigured haiku ceiling for coding is degraded to the sonnet floor rather than forwarded — and deliberately not to inherit, since the session model itself could be Haiku.

3

Desired tier: riskLevel == "high" OR complexity == "complex_feature" → desired = ceiling (hardest work gets the configured max). Otherwise desired = "sonnet" (down-route target for standard/simple, well-specified tasks).

4

Clamp to ceiling: actual = desired if rank[desired] ≤ rank[ceiling], else actual = ceiling. A ceiling is a hard cap, never exceeded — this branch is defensive/unreachable while sonnet stays rank 1, the minimum.

5

Reason strings for the audit log: "high-risk/complex → ceiling <m>" · "standard/simple → down-routed to sonnet" · "standard/simple, clamped to ceiling <m>" (the defensive branch).

WF2 Step 8

Per-task dispatch, escalate on struggle

  • Before each task's dispatch, Step 8 calls select_impl_model(<resolved implementation ceiling>, task.riskLevel, <Step-2 complexity>) and dispatches that task's subagent with the returned model.
  • Logged per task in session notes: task <id>: model <m> (<reason>) — makes over/under-routing auditable (AC3).
  • Escalate on struggle: the existing clean-state-restore-and-retry-once path now retries at the ceiling model, not the down-routed one, before falling through to normal Step 8 failure handling (AC2). One escalation, then normal handling — no infinite ladder.
  • When the ceiling is inherit, the block stays inert exactly as before: the function returns inherit and dispatch omits model:.
Hard guarantee

rawgentic never dispatches model: haiku

Three independent layers, so a single misconfiguration can't slip Haiku into coding work:

  • resolve() bumps any role — review, analysis, implementation — configured to haiku up to sonnet, with a stderr warning. This is a harder response than the soft opus floor: review: sonnet merely warns-and-applies, but review: haiku is overridden, not just flagged.
  • select_impl_model floors an unknown or haiku ceiling to sonnet (step 2 above) — and specifically not to inherit, because the session model under inherit could itself be Haiku.
  • Step 8's dispatch never emits model: haiku: if the session model is Haiku under inherit, dispatch uses sonnet instead.
