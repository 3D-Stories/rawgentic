# Brainstorming Session: Rawgentic Plugin Overhaul
**Date:** 2026-03-06 11:39 AM
**Goal:** Design a plan to improve ALL rawgentic skills, making them project-agnostic and config-driven via `.rawgentic.json`

## Phase 1: Project Context Exploration

### Current State Summary

**Plugin structure:**
- 10 skills: setup, create-issue, implement-feature, fix-bug, refactor, update-docs, update-deps, security-audit, optimize-perf, incident
- 1 hook: session-start (checks for CLAUDE.md setup marker)
- 1 template: claude-md-sdlc-section.md (used by setup to generate CLAUDE.md constants)
- Design docs for each workflow in docs/design/

**How configuration currently works:**
1. `/rawgentic:setup` runs, detects project environment, writes a "Project Constants" section into the project's CLAUDE.md
2. Constants are `${VARIABLE}` placeholders in CLAUDE.md that Claude "mentally substitutes" at runtime (prompt engineering, not real variable binding)
3. Each skill reads CLAUDE.md to find constants like `REPO`, `PROJECT_ROOT`, `TEST_COMMAND_UNIT`, etc.
4. Some skills also do filesystem probing (e.g., implement-feature checks for `.github/workflows/`, `migrations/`, etc.)

**Key problems identified by user:**
1. **Tech-stack bias** - Plugin was built assuming React/Node.js on Docker. Skills contain hardcoded references to npm, Docker Compose, specific file patterns
2. **No project folder awareness** - Skills assume they're run FROM the project root. No concept of a multi-project workspace (./projects/ folder pattern)
3. **Config in CLAUDE.md is fragile** - Constants embedded in CLAUDE.md via prompt engineering. No structured config file
4. **No project switching** - User runs Claude from a workspace with multiple projects in ./projects/. No mechanism to select/switch active project
5. **No blank project support** - Setup only handles existing projects. No wizard for brand-new projects

### User Requirements

1. `.rawgentic.json` config file per project (in project root, e.g., `./projects/rawgentic/.rawgentic.json`)
2. All skills read from `.rawgentic.json` instead of CLAUDE.md for constants
3. Skills must ask for project folder if not specified in invocation
4. Session-start hook should ask user which project to work on (suggest last used)
5. Setup should auto-detect tech stack for existing projects
6. Setup should brainstorm project intent for blank/new projects
7. Skills should be technology-agnostic (not React/Node.js specific)

## Phase 2: Clarifying Questions

### Q1: .rawgentic.json scope — replace or coexist with CLAUDE.md?
**Asked:** 2026-03-06 11:42 AM
**Options:**
- A) Full replacement of constants in .rawgentic.json; behavioral principles stay in CLAUDE.md
- B) Split — .rawgentic.json for constants, CLAUDE.md for principles + pointer
- C) Full migration — everything out of CLAUDE.md, pointer only
**Answer: C — Pointer only.** User rationale: switching between projects with different technologies means CLAUDE.md shouldn't hold project-specific anything. P1-P14 principles already live in skill files — they load when skills are invoked.

**Design decision:** CLAUDE.md rawgentic section is minimal:
```markdown
## Rawgentic
Active project: ./projects/<name>
Config: ./projects/<name>/.rawgentic.json
```
All constants, tech stack config, infrastructure details, test commands, deploy commands — everything goes in `.rawgentic.json`.

### Q2: .rawgentic.json schema depth — flat or structured?
**Asked:** 2026-03-06 11:47 AM
**Options:**
- A) Shallow + convention-based — techStack array, skills infer tools at runtime
- B) Deep + explicit — every tool, command, path explicitly mapped. No probing needed.
**Answer: B — Deep + explicit.** User rationale: skills shouldn't dig every time. Config is authoritative source of truth. Additionally, execution skills (implement-feature, fix-bug, etc.) should have instructions to UPDATE .rawgentic.json when they discover new project capabilities.

**Design decision:** .rawgentic.json uses deep schema with explicit mappings for:
- testing (frameworks, commands, config files)
- database (type, cli, container, migrations dir)
- security (auth mechanisms, middleware paths)
- services (name, port, framework per service)
- infrastructure (hosts, ssh users, docker compose paths)
- deploy (method, commands)

Skills that discover new capabilities during execution should update .rawgentic.json — creating a "learning config" pattern.

**Proposal:** `.rawgentic_workspace.json` at Claude root with activeProject, lastUsed, recentProjects[]

- Any skill invocation without explicit project reference reads activeProject from workspace file
- Skills that specify a project folder override the workspace default (and update it)

### Q4: Blank project setup + workspace management
**Asked:** 2026-03-06 11:55 AM
**Original question:** How much scaffolding for blank projects?
**User response:** Expanded scope significantly. Instead of just handling blank projects in setup, create a proper workspace management layer.

**Design decisions:**

**`.rawgentic_workspace.json` is now a project registry:**
```json
{
  "projects": [
    { "name": "my-app", "path": "./projects/my-app", "active": true, "lastUsed": "2026-03-06T..." },
    { "name": "rawgentic", "path": "./projects/rawgentic", "active": false, "lastUsed": "2026-03-05T..." }
  ]
}
```

**New skill: `rawgentic:new-project`** — Two paths:
1. **No folder exists** -> create `./projects/<name>`, add to workspace registry, then:
   - Ask if pre-existing GitHub repo -> `git clone` into folder -> run `rawgentic:setup` (existing project flow)
   - Or brand new -> run `rawgentic:setup` (blank project flow with brainstorm)
2. **Folder exists** -> add to workspace registry -> run `rawgentic:setup` (existing project flow)

**Updated `rawgentic:switch`** — Two invocation styles:
- `/rawgentic:switch my-app` — by registered project name
- `/rawgentic:switch ./projects/my-app` — by path
- Sets `active: true` on target, `active: false` on all others

**Updated `rawgentic:setup`** — Purely about configuring `.rawgentic.json`. Two sub-flows:
- Existing code detected -> tech stack detection, populate deep config
- Empty/new project -> brainstorm intent, create skeleton `.rawgentic.json`

**Separation of concerns:**
- **Workspace layer:** `.rawgentic_workspace.json`, `new-project`, `switch` — manages WHICH project
- **Config layer:** `.rawgentic.json`, `setup` — manages HOW a project is configured
- **Runtime:** all other skills read active project's `.rawgentic.json`

## Phase 3: Design (presented section by section)

### Section 1: `.rawgentic.json` Schema -- APPROVED

```json
{
  "project": {
    "name": "string",
    "type": "application | infrastructure | scripts | library | docs | research",
    "description": "string",
    "root": "."
  },
  "repo": {
    "provider": "github",
    "fullName": "org/repo",
    "defaultBranch": "main"
  },
  "techStack": ["string"],
  "testing": {
    "frameworks": [
      {
        "name": "string",
        "type": "unit | e2e | integration",
        "command": "string",
        "configFile": "string (optional)",
        "testDir": "string (optional)"
      }
    ]
  },
  "database": {
    "type": "string (optional)",
    "cli": "string (optional)",
    "name": "string (optional)",
    "user": "string (optional)",
    "container": "string (optional)",
    "migrationsDir": "string (optional)",
    "migrationPattern": "string (optional)"
  },
  "services": [
    {
      "name": "string",
      "type": "frontend | backend | worker | api",
      "framework": "string",
      "port": "number",
      "entryPoint": "string (optional)",
      "healthCheck": "string (optional)"
    }
  ],
  "infrastructure": {
    "hosts": [
      {
        "name": "string",
        "address": "string",
        "sshUser": "string",
        "description": "string (optional)"
      }
    ],
    "docker": {
      "composeFiles": [
        { "name": "string", "path": "string" }
      ]
    }
  },
  "deploy": {
    "method": "compose | script | ssh | manual",
    "command": "string (optional)",
    "targetHost": "string (optional)"
  },
  "security": {
    "authMechanisms": [
      {
        "type": "string",
        "appliesTo": ["string"],
        "middlewarePath": "string (optional)",
        "headerName": "string (optional)"
      }
    ],
    "validationLibrary": "string (optional)",
    "dataChannels": ["string"]
  },
  "ci": {
    "provider": "string (optional)",
    "workflowDir": "string (optional)"
  },
  "formatting": {
    "tool": "string (optional)",
    "configFile": "string (optional)",
    "command": "string (optional)"
  },
  "documentation": {
    "primaryFiles": ["string"],
    "format": "string"
  }
}
```

All sections optional except `project` and `repo`. Skills adapt behavior based on what's present.

### Section 2: `.rawgentic_workspace.json` Schema -- APPROVED

```json
{
  "version": 1,
  "projectsDir": "./projects",
  "projects": [
    {
      "name": "string",
      "path": "string (relative to Claude root)",
      "active": "boolean (exactly one true)",
      "lastUsed": "ISO 8601 timestamp",
      "configured": "boolean (true after setup completes)"
    }
  ]
}
```

**Invariants:**
- If projects.length > 0, exactly one has active: true
- new-project adds entry and sets it active (deactivates previous)
- switch sets target active (deactivates previous)
- setup sets configured: true on completion
- lastUsed updated whenever a skill runs against that project

### Section 3: Session-start Hook -- APPROVED

**Bash script reads `.rawgentic_workspace.json`, injects context via `hookSpecificOutput.additionalContext`.**

3-state flow:
1. No workspace file -> suggest `/rawgentic:new-project`
2. No projects registered -> suggest `/rawgentic:new-project`
3. Active project found -> report name + path. If not configured -> suggest `/rawgentic:setup`

No interactive prompting, no project switching, no reading `.rawgentic.json`. Pure context injection.

### Section 4: Config-loading Protocol -- APPROVED

Inline `<config-loading>` block in every workflow skill:
1. Read `.rawgentic_workspace.json` -> find active project (STOP if missing)
2. Read `<activeProject.path>/.rawgentic.json` -> parse to `config` (STOP if missing)
3. Build `capabilities` object from config (has_tests, has_ci, has_deploy, etc.)
4. Update `lastUsed` timestamp in workspace file
5. Skills that discover new capabilities update `.rawgentic.json` before completing

Hard stop on missing config. No filesystem probing for config-level data. Learning config pattern enforced.

### Section 5: Skill Designs (setup, new-project, switch) -- APPROVED

**5a: `rawgentic:setup` (rewrite)**
7-step wizard:
1. Verify context (active project from workspace)
2. Detect (existing code -> auto-detect tech stack) or brainstorm (blank -> /superpowers:brainstorm for intent)
3. Present detected config as filled .rawgentic.json
4. User confirms/edits each section
5. Write .rawgentic.json to project root
6. Update workspace (configured: true)
7. Verify (file written, repo accessible)

Key: No CLAUDE.md writing. Detects auth, services, data channels — not just basic tech stack.

**5b: `rawgentic:new-project` (new)**
5-step flow:
1. Parse input (name or path)
2. Check if already registered (offer switch or re-setup)
3. Folder check:
   - Doesn't exist -> create, ask about GitHub clone vs brand new, git init
   - Exists -> proceed
4. Register in workspace (active: true, configured: false)
5. Delegate to rawgentic:setup

**5c: `rawgentic:switch` (new)**
4-step flow:
1. Parse input (name or path, 4 invocation styles)
2. Find in workspace (by name then path, suggest new-project if not found)
3. Switch active flag + update lastUsed
4. Confirm (report name, path, configured status)
