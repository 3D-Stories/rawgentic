## Analyst Observations

### Strongly Discriminating Scenarios (skill provides clear measurable value)

1. **implement-feature/config-contradicts-filesystem** (1.0 vs 0.0) — R5 test. With-skill followed config (Implement-Verify mode), without-skill trusted filesystem and wrote Jest tests. Best R5 demonstration.

2. **implement-feature/config-missing-rawgentic-json** (1.0 vs 0.0) — Error path. With-skill stopped and referenced `/rawgentic:setup`, without-skill proceeded without config.

3. **security-audit/learning-config-update** (1.0 vs 0.0) — R4 test. With-skill appended API-Key to authMechanisms[] and wrote config back. Without-skill only recommended the change.

4. **update-deps/security-first-sentinel** (1.0 vs 0.0) — With-skill demonstrated security-first ordering with sentinel test command. Without-skill described methodology but never simulated it.

5. **refactor/characterization-tests-sentinel** (1.0 vs 0.0) — With-skill wrote 12 characterization tests using sentinel vitest command. Without-skill halted without source code.

6. **refactor/no-tests-degradation** (1.0 vs 0.5) — With-skill provided 5-strategy adaptation framework. Without-skill acknowledged but didn't articulate preservation strategy.

### Non-Discriminating Scenarios (baseline also passes)

17/25 scenarios showed no measurable difference. Root causes:

- **Config reading is baseline competence**: Modern LLMs naturally read JSON config files when pointed at them. Sentinel value assertions pass for both runs since both read the config.
- **Sandbox limitations**: Without real source code/infrastructure, both runs produce simulated workflows. Assertions check config presence, not workflow execution quality.
- **CLAUDE.md contradiction was too obvious**: The "DELIBERATELY WRONG" annotation made it trivial for the baseline to ignore CLAUDE.md.
- **Adaptation scenarios are natural**: When config says "no tests," both skill and no-skill agents naturally skip test steps.

### Recommendations for Iteration 2

1. **Add workflow-structure assertions**: Test for step markers, quality gates, critique outputs — behaviors unique to the skill workflows.
2. **Add negative assertions**: "Does NOT use npm test" (only vitest --sentinel), "Does NOT target README.md" (only sentinel paths).
3. **Make CLAUDE.md contradiction realistic**: Remove the "DELIBERATELY WRONG" annotation. Use a plausible but incorrect repo name.
4. **Add source code to fixtures**: Even minimal mock files would let agents actually execute (or simulate more convincingly), making assertions more discriminating.
5. **Test learning-config more broadly**: The 1.0 vs 0.0 result on security-audit/learning-config-update proves this is the most discriminating assertion type. Add it to more skills.

## Integration Test: setup-to-implement-feature

Tests whether a `.rawgentic.json` generated by the setup skill can be loaded by implement-feature without schema errors.

| Run | config-loads-successfully | schema-compatible | Pass Rate |
|-----|--------------------------|-------------------|-----------|
| with_skill | PASS | PASS | 100% (2/2) |
| without_skill | FAIL | PASS | 50% (1/2) |

**Key finding**: The setup-generated config is minimal (4/10 sections present) but fully compatible. Missing sections (`testing`, `ci`, `deploy`, `database`, `infrastructure`, `formatting`) resolve to `false` capability flags — graceful degradation, not errors. The without_skill agent parsed the config but never built a capabilities object, failing the first assertion.

**Cross-skill schema compatibility confirmed.** No schema drift between setup and implement-feature.
